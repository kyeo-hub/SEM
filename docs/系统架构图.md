## 0. 角色权限架构

### 0.1 角色定义

系统定义 3 种角色，每种角色有不同的权限范围：

| 角色 | 标识 | 说明 | 典型用户 |
|------|------|------|----------|
| **管理员** | `admin` | 拥有系统全部权限 | 设备管理员、系统管理员 |
| **维保员** | `maintainer` | 负责设备维保、点检 | 维修工、保养人员 |
| **操作司机** | `operator` | 负责设备作业、点检 | 起重机司机、操作手 |

### 0.2 权限矩阵

```
┌────────────────┬──────────┬────────────┬──────────────┐
│ 操作类型       │ 管理员   │ 维保员     │ 操作司机     │
├────────────────┼──────────┼────────────┼──────────────┤
│ 作业           │ ✅       │ ❌         │ ✅           │
│ 待命           │ ✅       │ ✅         │ ✅           │
│ 点检           │ ✅       │ ✅         │ ✅           │
│ 维保           │ ✅       │ ✅         │ ❌           │
│ 故障登记       │ ✅       │ ✅         │ ✅           │
│ 用户管理       │ ✅       │ ❌         │ ❌           │
│ 设备管理       │ ✅       │ ❌         │ ❌           │
│ 点检标准       │ ✅       │ ❌         │ ❌           │
└────────────────┴──────────┴────────────┴──────────────┘

核心规则:
- 操作司机：可以点检但不能维保
- 维保员：可以点检但不能作业
- 管理员：拥有全部权限
```

## 2. 数据流架构

```mermaid
sequenceDiagram
    participant User as 用户
    participant Web as Web 前端
    participant API as Go API
    participant DB as PostgreSQL
    participant Redis as Redis
    participant WeChat as 企业微信

    User->>Web: 操作请求
    Web->>API: HTTPS/API 调用
    API->>Redis: 检查缓存
    alt 缓存命中
        Redis-->>API: 返回数据
    else 缓存未命中
        API->>DB: 查询数据库
        DB-->>API: 返回数据
        API->>Redis: 写入缓存
    end
    API-->>Web: 返回响应
    Web-->>User: 展示结果

    Note over API,WeChat: 故障/维修事件
    API->>WeChat: Webhook 推送
    WeChat-->>API: 推送结果
```

## 3. 点检业务流程

```mermaid
flowchart TD
    Start([开始点检]) --> Scan[扫码/选择设备]
    Scan --> Info[获取设备信息]
    Info --> Shift{选择班次}
    
    Shift -->|班前 | Form[点检表单]
    Shift -->|班中 | Form
    Shift -->|交班 | Form
    
    Form --> Check{逐项点检}
    Check -->|正常 | Next[下一项]
    Check -->|异常 | Record[记录异常 + 拍照]
    Record --> Next
    
    Next --> More{还有项目？}
    More -->|是 | Check
    More -->|否 | Problem[填写问题记录]
    
    Problem --> Sign[电子签名]
    Sign --> Submit[提交点检]
    Submit --> Save[保存数据库]
    Save --> Notify{有异常？}
    
    Notify -->|是 | Push[推送告警]
    Notify -->|否 | Direct[直接完成]
    
    Push --> End([完成])
    Direct --> End
    
    style Start fill:#90EE90
    style End fill:#87CEEB
    style Notify fill:#FFD700
```

## 5. 统一二维码设计 (4 个操作选项)

```mermaid
flowchart LR
    QR[统一二维码<br/>设备 UUID] --> Scan[手机浏览器扫码]
    Scan --> H5[H5 移动端设备操作页]

    H5 --> Info[获取设备信息<br/>展示状态/船名/货品]

    Info --> Action{选择操作类型}

    Action -->|🟢 作业 | Op1[开始作业<br/>填写船名/货品]
    Action -->|🔵 待命 | Op2[结束作业<br/>确认时长]
    Action -->|📋 点检 | Op3[点检录入<br/>选择班次→表单]
    Action -->|🔧 维保 | Op4[维保登记<br/>类型/内容/故障等级]

    Op1 --> Status1[状态→working<br/>记录开始时间<br/>大屏更新]
    Op2 --> Status2[状态→standby<br/>记录结束时间<br/>生成历史]
    Op3 --> Status3[提交点检记录<br/>异常→故障等级<br/>推送企微]
    Op4 --> Status4[状态→maintenance/fault<br/>推送企微通知]

    style QR fill:#000000,color:#ffffff
    style H5 fill:#1890ff,color:#ffffff
    style Op1 fill:#90EE90
    style Op2 fill:#87CEEB
    style Op3 fill:#FFD700
    style Op4 fill:#FFA500
```

**说明**:
- 每台设备一个统一二维码，包含设备 UUID
- 扫码后进入 H5 移动端设备操作页，展示设备当前状态
- 4 个操作选项：作业 (开始作业)、待命 (结束作业)、点检 (点检录入)、维保 (维保登记)
- 故障报告不是独立选项，而是在点检异常或维保登记时选择故障等级
- **技术变更**: 使用 H5 移动端替代微信小程序，通过 html5-qrcode 库实现扫码

## 8. 文件上传流程

```mermaid
sequenceDiagram
    participant User as 用户
    participant Client as 前端
    participant API as Go API
    participant Validator文件验证
    participant Storage对象存储
    participant DB as 数据库

    User->>Client: 选择/拍摄图片
    Client->>API: 上传请求
    API->>Validator: 验证文件 (类型/大小)
    Validator-->>API: 验证通过
    
    alt 本地存储
        API->>Storage: 保存到本地文件系统
    else OSS 存储
        API->>Storage: 上传到阿里云 OSS
    end
    
    Storage-->>API: 返回文件 URL
    API->>Storage: 生成缩略图
    Storage-->>API: 缩略图 URL
    
    API->>DB: 保存文件记录
    DB-->>API: 记录 ID
    
    API-->>Client: 返回文件信息
    Client-->>User: 显示预览
    
    Note over API,DB: 关联到点检记录/明细
```

## 9. 认证授权架构

```mermaid
flowchart TD
    subgraph WebAuth["Web 认证流程"]
        User1[用户] --> Login1[登录页面]
        Login1 --> Credentials1[用户名/密码]
        Credentials1 --> API1[Go API 验证]
        API1 --> JWT1[生成 JWT Token]
        JWT1 --> Storage1[存入 Redis]
        JWT1 --> Return1[返回前端]
        Return1 --> Header1[存入 localStorage]
        Header1 --> Request1[后续请求携带]
        Request1 --> Middleware1[JWT 中间件验证]
        Middleware1 --> Access1[访问资源]
    end

    subgraph MiniAppAuth["小程序认证流程"]
        User2[用户] --> Launch2[打开小程序]
        Launch2 --> Login2[微信登录]
        Login2 --> Code2[获取 Code]
        Code2 --> API2[Go API]
        API2 --> WeChatAPI[微信 code2session]
        WeChatAPI --> Session2[获取 OpenID]
        Session2 --> JWT2[生成 JWT Token]
        JWT2 --> Return2[返回小程序]
        Return2 --> Storage2[存入 Storage]
        Storage2 --> Request2[后续请求携带]
        Request2 --> Middleware2[JWT 中间件验证]
        Middleware2 --> Access2[访问资源]
    end

    style WebAuth fill:#e1f5ff
    style MiniAppAuth fill:#fff4e1
```
